CHAPTER 1 · KUBERNETES

Note Canva on Kubernetes

Everything from zero to a production GKE deployment — with GitHub Actions CI/CD, autoscaling, zero-downtime deploys, and a full ops reference. You know Docker Compose. That's exactly the right starting point.

Stack: Node.js · GKE · Redis · PostgreSQL · BullMQ
Region: europe-west1
Services: api · realtime · worker

Architecture overview

Three services, one public IP. Traffic enters through a Google Cloud Load Balancer, hits a Kubernetes Ingress that routes by URL path, and fans out to the right pods. Think of it like your Nginx config — but managed, auto-scaling, and self-healing.

Docker Compose → Kubernetes

Your brain already knows this

You know Docker Compose. Kubernetes is the same idea — declare what you want to run, and a control plane makes it happen. The difference: Compose manages one machine. K8s manages a fleet of machines, keeps your declared count alive automatically, rolls out updates with zero downtime, and self-heals when pods crash.

Think of a Deployment like a React component spec, and a Pod like an actual rendered instance. The Deployment says "I want 3 of these." K8s makes sure there are always 3, replacing any that crash.

Memorystore Redis ×2bash
# Realtime pub/sub — latency-critical
gcloud redis instances create redis-realtime \
  --size=1 --region=europe-west1 \
  --tier=STANDARD_HA --redis-version=redis_7_0

# BullMQ job queues — kept separate deliberately
gcloud redis instances create redis-jobs \
  --size=1 --region=europe-west1 \
  --tier=STANDARD_HA --redis-version=redis_7_0
Why two separate Redis instances?

Same reason you don't run your main thread and a Web Worker in the same thread. BullMQ queues can spike under load — if they share Redis with your realtime pub/sub, a flood of queued jobs adds latency to every WebSocket message. Separate instances = isolated failure domains.

2 Config philosophy — what goes where

The one rule you need

If this value leaked publicly, would you need to rotate credentials? Yes → Secret. No → ConfigMap.

Never put a password and a hostname in the same object. Never embed a password inside a connection URL string — you can't rotate just the password without rebuilding the whole string. Keep them separate.

ConfigMap — safe to reference in scriptsSecret — CLI only, never committed
NODE_ENV=production, LOG_LEVEL=infoDB_PASSWORD
DB_HOST, DB_PORT, DB_NAME, DB_USERJWT_SECRET
REDIS_REALTIME_HOST, REDIS_JOBS_HOSTSENDGRID_API_KEY
WORKER_PREVIEW_CONCURRENCY=8REDIS_AUTH (if Redis auth enabled)

2 ConfigMap — GCP IPs fetched at deploy time

Redis and DB host values are private IPs that live in GCP. A script fetches them live from GCP at deploy time and writes them into K8s. Your repo contains the script — not the values.

scripts/configure-cluster.shbash
#!/bin/bash
set -e
PROJECT=note-canva-prod
REGION=europe-west1

DB_HOST=$(gcloud sql instances describe note-canva-db \
  --project=$PROJECT --format='value(ipAddresses[0].ipAddress)')

REDIS_REALTIME_HOST=$(gcloud redis instances describe redis-realtime \
  --region=$REGION --project=$PROJECT --format='value(host)')

REDIS_JOBS_HOST=$(gcloud redis instances describe redis-jobs \
  --region=$REGION --project=$PROJECT --format='value(host)')

kubectl create configmap backend-config \
  --namespace=prod \
  --from-literal=NODE_ENV=production \
  --from-literal=LOG_LEVEL=info \
  --from-literal=DB_HOST=$DB_HOST \
  --from-literal=DB_PORT=5432 \
  --from-literal=DB_NAME=notecanva \
  --from-literal=DB_USER=api_user \
  --from-literal=REDIS_REALTIME_HOST=$REDIS_REALTIME_HOST \
  --from-literal=REDIS_JOBS_HOST=$REDIS_JOBS_HOST \
  --from-literal=REDIS_PORT=6379 \
  --from-literal=WORKER_PREVIEW_CONCURRENCY=8 \
  --dry-run=client -o yaml | kubectl apply -f -

echo "ConfigMap updated ✓"

2 Secrets — per-service, created via CLI

Each service gets only the secrets it actually needs. API never sees SENDGRID_API_KEY. Worker never sees JWT_SECRET. Least-privilege: a compromised service can't leak unrelated credentials.

scripts/create-secrets.shbash
#!/bin/bash
# Values entered interactively — never stored in this file
read -s -p "DB_PASSWORD: " DB_PASS;   echo
read -s -p "JWT_SECRET: "  JWT;       echo
read -s -p "SENDGRID_KEY: " SENDGRID; echo

kubectl create secret generic api-secrets --namespace=prod \
  --from-literal=DB_PASSWORD=$DB_PASS \
  --from-literal=JWT_SECRET=$JWT \
  --dry-run=client -o yaml | kubectl apply -f -

kubectl create secret generic worker-secrets --namespace=prod \
  --from-literal=DB_PASSWORD=$DB_PASS \
  --from-literal=SENDGRID_API_KEY=$SENDGRID \
  --dry-run=client -o yaml | kubectl apply -f -

kubectl create secret generic realtime-secrets --namespace=prod \
  --from-literal=JWT_SECRET=$JWT \
  --dry-run=client -o yaml | kubectl apply -f -

echo "Secrets created ✓"

3 api service

k8s/prod/api-deployment.yamlyaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
  namespace: prod
spec:
  replicas: 2
  selector:
    matchLabels:
      app: api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0   # never kill a pod before new one is ready — zero downtime
      maxSurge: 1         # spin up 1 extra pod during rollout
  template:
    metadata:
      labels:
        app: api
    spec:
      terminationGracePeriodSeconds: 30
      containers:
        - name: api
          image: europe-west1-docker.pkg.dev/note-canva-prod/note-canva/api:latest
          ports:
            - containerPort: 8080
          envFrom:
            - configMapRef:
                name: backend-config
            - secretRef:
                name: api-secrets
          env:
            - name: POD_NAME
              valueFrom:
                fieldRef:
                  fieldPath: metadata.name
          resources:
            requests:
              cpu: "250m"    # 0.25 cores — guaranteed minimum
              memory: "512Mi"
            limits:
              cpu: "1000m"  # 1 core — throttled if exceeded
              memory: "1Gi"  # OOMKilled if exceeded
          readinessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
            failureThreshold: 3
          livenessProbe:
            httpGet:
              path: /livez
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 20
            failureThreshold: 3
---
apiVersion: v1
kind: Service
metadata:
  name: api
  namespace: prod
spec:
  selector:
    app: api
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP
Readiness vs Liveness — they're different!

readinessProbe = "am I ready to receive traffic?" K8s won't send requests to a pod until this passes. Check your DB connection here. If DB is down → return 503 → K8s routes around this pod.

livenessProbe = "am I still alive at all?" K8s restarts the pod if this fails 3× in a row. Keep this dead simple — never check external deps. A slow DB shouldn't cause a pod restart.

src/routes/health.tstypescript
// readiness: checks DB — if DB is unreachable, stop routing here
app.get('/healthz', async (req, res) => {
  try {
    await db.query('SELECT 1')
    res.json({ status: 'ok', pod: process.env.POD_NAME })
  } catch {
    res.status(503).json({ status: 'db_unreachable' })
  }
})

// liveness: always returns ok — just proves the process is alive
app.get('/livez', (_req, res) => {
  res.json({ status: 'alive' })
})

3 realtime service (StatefulSet)

Uses a StatefulSet instead of a Deployment. StatefulSet pods get stable, predictable names — realtime-0, realtime-1 — that survive restarts. Your sharding code reads the ordinal from HOSTNAME, which Kubernetes sets automatically.

k8s/prod/realtime-statefulset.yaml (key parts)yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: realtime
  namespace: prod
spec:
  serviceName: realtime
  replicas: 2
  template:
    spec:
      terminationGracePeriodSeconds: 30
      containers:
        - name: realtime
          image: europe-west1-docker.pkg.dev/note-canva-prod/note-canva/realtime:latest
          env:
            # HOSTNAME is set automatically: "realtime-0", "realtime-1" etc.
            # DO NOT declare it — just read it in your code
            - name: TOTAL_REALTIME_PODS
              value: "2"   # keep in sync with replicas
          resources:
            requests: { cpu: "300m", memory: "512Mi" }
            limits:   { cpu: "1500m", memory: "1Gi" }

3 worker service

No Service object — worker pods pull jobs from Redis queues, they don't receive inbound traffic. Note the longer terminationGracePeriodSeconds: 60 — workers need extra time to finish in-progress jobs.

4 Graceful shutdown — required in every service

Don't skip this

When K8s stops a pod — during a deploy, scale-down, or node drain — it sends SIGTERM. If you ignore it, in-flight requests and jobs get dropped mid-execution. This is the backend equivalent of not cleaning up event listeners before unmounting a React component.

src/api/index.ts + src/worker/index.tstypescript
// API: stop accepting connections, drain in-flight requests
process.on('SIGTERM', async () => {
  server.close()
  await new Promise(r => setTimeout(r, 25_000))  // let requests finish
  await db.end()
  process.exit(0)
})

// Worker: pause queue consumption, finish current jobs
process.on('SIGTERM', async () => {
  await previewWorker.pause()
  await mailWorker.pause()
  await Promise.all([previewWorker.close(), mailWorker.close()])
  process.exit(0)
})

5 GitHub CI/CD — how it works

Push to main

GitHub Actions triggers on every push or PR merge to main.

Build all three Docker images

Tagged with the full git commit SHA — unique per commit, auditable forever.

Push to Artifact Registry

Auth via Workload Identity Federation — no service account key files to rotate or leak.

kubectl set image

Updates each Deployment to the new SHA tag. K8s starts rolling update: one new pod starts, passes readiness probe, then one old pod stops.

Wait for rollout

kubectl rollout status blocks until all pods are healthy. Workflow fails if it takes more than 5 minutes.

5 GitHub Actions workflow

.github/workflows/deploy.ymlyaml
name: Deploy to GKE
on:
  push:
    branches: [main]
env:
  REGISTRY: europe-west1-docker.pkg.dev/${{ secrets.GCP_PROJECT_ID }}/note-canva
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - name: Auth to GCP
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.WIF_SERVICE_ACCOUNT }}
      - uses: google-github-actions/setup-gcloud@v2
      - name: Configure Docker
        run: gcloud auth configure-docker europe-west1-docker.pkg.dev --quiet
      - uses: google-github-actions/get-gke-credentials@v2
        with:
          cluster_name: ${{ secrets.GKE_CLUSTER_NAME }}
          location: ${{ secrets.GCP_REGION }}
      - name: Build & push
        run: |
          TAG=${{ github.sha }}
          docker build -f Dockerfile.api      -t $REGISTRY/api:$TAG      . && docker push $REGISTRY/api:$TAG
          docker build -f Dockerfile.realtime -t $REGISTRY/realtime:$TAG . && docker push $REGISTRY/realtime:$TAG
          docker build -f Dockerfile.worker   -t $REGISTRY/worker:$TAG   . && docker push $REGISTRY/worker:$TAG
      - name: Deploy
        run: |
          TAG=${{ github.sha }}
          kubectl set image deployment/api       api=$REGISTRY/api:$TAG       -n prod
          kubectl set image statefulset/realtime realtime=$REGISTRY/realtime:$TAG -n prod
          kubectl set image deployment/worker    worker=$REGISTRY/worker:$TAG  -n prod
          kubectl rollout status deployment/api       -n prod --timeout=5m
          kubectl rollout status statefulset/realtime -n prod --timeout=5m
          kubectl rollout status deployment/worker    -n prod --timeout=5m

↗ Interview questions — Kubernetes

Senior · system design · asked at Spotify, Zalando, Delivery Hero
"What's the difference between a Deployment and a StatefulSet, and when would you use each?"
A Deployment manages stateless pods — they're interchangeable. Pods get random names like api-7d9f8b-xkp2z. When one dies and restarts, it gets a different name and potentially a different node. This is fine for REST APIs where all instances are identical.

A StatefulSet gives pods stable, predictable identities: realtime-0, realtime-1. These names persist across restarts. You use this when pod identity matters — for sharding (pod-0 owns shard 0, pod-1 owns shard 1), for leader election, or for distributed systems that need stable network addresses. The trade-off: StatefulSet rolling updates are sequential (one pod at a time), which is slower but safer for stateful workloads.
Senior · operations · classic pod failure scenario
"A pod is in CrashLoopBackOff. Walk me through how you'd debug it."
1. kubectl logs POD -n prod --previous — the previous flag shows logs from the crashed container instance, before it was restarted. This usually contains the actual error.

2. kubectl describe pod POD -n prod — look at the Events section at the bottom. K8s records why it killed the pod (OOMKilled, liveness probe failure, etc.) and what happened before that.

3. kubectl exec deployment/api -n prod -- env | grep DB_ — if the logs show a connection error, verify the env vars are actually set correctly. Secrets might not have been created or have wrong values.

The most common root causes in order: wrong env var / missing secret, app failing to connect to DB/Redis on startup, OOMKilled (memory limit too low), bad image tag / image never built.
Mid-level · conceptual · asked constantly
"What's the difference between readiness and liveness probes?"
Readiness: "should K8s send traffic to this pod?" Fails → pod removed from Service load balancer. Used to signal "I'm still starting up" or "my upstream dependency is down." Check external deps here.

Liveness: "is this pod still alive at all?" Fails 3× → pod restarted. Used to detect a deadlock or infinite loop where the process is running but hung. Never check external deps here — a slow DB shouldn't restart your pod, it should just stop routing traffic to it.
Staff · architecture · scaling discussion
"You've set minReplicas: 2 on your HPA. Why not minReplicas: 1?"
Single replica = single point of failure. During a rolling update, K8s terminates the old pod before (or during) starting the new one. With 1 replica, there's a window where zero pods are running. With 2 replicas + maxUnavailable: 0, at least 1 pod is always alive during the update.

Additionally, with 1 replica, a node drain (GCP maintenance, node upgrade) would take your service down. With 2 replicas on different nodes (K8s schedules them apart by default), a node going down loses 1 pod but traffic keeps flowing to the other.

The extra cost of a second pod is the cheapest SLA insurance you'll ever buy.

T gcloud vs Terraform — the real difference

Your brain already knows this

gcloud commands = jQuery. Imperative. You tell it exactly what to do each time. If the script fails halfway, you don't know what exists.

Terraform = React. Declarative. You describe desired state. Terraform diffs current state against desired state and applies only the minimum change. Running it twice is safe. This is idempotency — and it's the reason infrastructure-as-code works at scale.

Concerngcloud CLITerraform
ReproducibilityScript breaks if state driftsAlways converges to declared state
Review processWho ran what, when? UnknownEvery change is a PR + plan output
Multi-envCopy-paste scripts, manual editsModules + variable files
TeardownManual, miss something, costs moneyterraform destroy — total cleanup
Drift detectionNo visibilityterraform plan shows infra vs code delta

The mental model — three concepts, that's it

State — Terraform's memory of what exists

A terraform.tfstate file maps your configuration to real cloud resources. If someone manually deletes a resource, terraform plan detects the drift. Store state remotely in a GCS bucket — never locally in a team environment.

Plan — the diff engine

Before any change, Terraform computes the delta between current state and desired state. Review it, approve it, then apply. In CI, post the plan output as a PR comment so reviewers see exactly what infra will change alongside the code change.

Modules — reusable infra components

A module is a folder of .tf files encapsulating a logical unit — "a GKE cluster", "a Cloud SQL instance". Call a module like a function: parameters in, resources out. This is how you provision identical infra in multiple regions without copy-paste.

Don't have one flat directory with prod.tf and staging.tf sharing a single state file. A failed staging apply will lock prod. Use separate state per environment — isolated blast radiuses.

T2 Multi-region — why it's not just "run it twice"

The hard truth about distributed systems

Running your app in two regions solves one problem (regional outage) and introduces five others: data consistency, latency routing, split-brain, failover coordination, and cost doubling.

Think of it as the backend equivalent of state management at scale. A user in Tokyo and a user in London are like two browser tabs with their own local state — they need to eventually sync. The question isn't "can I run in two regions?" The question is "which data can I afford to have stale, and for how long?" That's your consistency model, and it drives every decision.

TopologyHow it worksBest forRisk
Active/PassiveAll writes go to primary. Secondary is warm standby. Manual or auto failover.Most apps. Simple operational model.1–5 min failover. Small data loss window.
Active/Active (reads)Writes go to primary. Reads served from any region via read replicas.Read-heavy apps: dashboards, contentReplica lag — reads may be slightly stale
Active/Active (writes)Any region accepts writes. Conflict resolution required.Real-time collaboration, global fintechExtremely complex. Needs CRDT-aware data model or Spanner/CockroachDB.

For Note Canva today: start with Active/Passive. You get 99.9% of the reliability benefit at 20% of the complexity.

The replication lag question — you WILL be asked this

Cross-region replication is asynchronous. There is always a lag — typically milliseconds, but can spike. If primary fails, the replica may be N seconds behind. This is your RPO (Recovery Point Objective). For most apps, losing a few seconds of writes is acceptable. For financial transactions, it is not. Know your RPO before choosing a replication topology.

Failover walkthrough

Staff-level · distributed systems · asked at Cloudflare, Stripe, N26
"Your primary region (europe-west1) goes down. Walk me through what happens in the first 5 minutes."
T+0s: Health check probes start failing against europe-west1 backends. The Global LB's outlier detection begins counting consecutive errors.

T+30s: europe-west1 backends ejected from the LB pool. New traffic routes automatically to us-central1. Users see elevated latency — not an outage.

T+2min: Ops team alerted via PagerDuty. Decision point: transient spike, or actual regional failure?

T+5min: If confirmed regional failure: promote the Cloud SQL read replica in us-central1 to primary. Update Terraform state. us-central1 is now fully read/write.

Key insight: Application traffic failover is automatic and sub-minute. Database failover is manual and takes 2–5 minutes. This is your RTO. Design your app to tolerate brief DB unavailability with retry loops and connection pool circuit breakers.

T3 GitOps — the model shift

The key insight

The CI/CD pipeline in Chapter 1 is a push model: GitHub Actions runs kubectl apply directly. This works for one cluster and 3 engineers. At scale it breaks down — your cluster state is only as reliable as the last successful CI run.

GitOps flips this. A controller running inside your cluster watches a Git repository and reconciles toward it continuously. Git is the single source of truth. Manual kubectl apply in prod is now an anti-pattern — like mutating the DOM directly in a React app.

T4 Secrets at scale — three layers

GCP Secret Manager — source of truth

All secrets live here. Versioned. IAM-controlled. Every access is logged in Cloud Audit Logs. Rotation is built-in. You never see the raw value in your cluster YAML.

Workload Identity — keyless authentication

Pods authenticate to GCP as a service account without needing a key file. A K8s service account is federated to a GCP service account. The pod gets a short-lived token automatically. No secrets to rotate for auth itself.

External Secrets Operator — the bridge

Reads from Secret Manager and writes into K8s Secrets automatically. Your pods still consume standard env.valueFrom.secretKeyRef — nothing in application code changes. Sync happens externally.

Workload Identity binding

Bind K8s service account to GCP service accountbash
# 1. Create GCP service account for api pods
gcloud iam service-accounts create note-canva-api \
  --display-name="Note Canva API pods"

gcloud projects add-iam-policy-binding note-canva-prod \
  --member="serviceAccount:note-canva-api@note-canva-prod.iam.gserviceaccount.com" \

gcloud iam service-accounts add-iam-policy-binding \

SLOs and error budgets

The concept that changes how you talk about reliability

An SLO (Service Level Objective) is a target: "99.9% of API requests complete in under 500ms." An error budget is what you're allowed to burn: 0.1% of requests = ~43 minutes of downtime per month. If you have budget left, you can deploy. If you're burning through it, you freeze deployments.

This removes politics from "is it safe to ship?" — the answer is in the numbers.

Network Policy — zero-trust inside the cluster

k8s/prod/network-policy.yaml — deny all, then allow explicitlyyaml
# 1. Default deny-all ingress in prod namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: prod
spec:
  podSelector: {}  # matches ALL pods
  policyTypes: [Ingress]
---
# 2. Allow only ingress controller → api pods
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-ingress-to-api
  namespace: prod
spec:
  podSelector:
    matchLabels:
      app: api
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
      ports:
        - port: 3000

Supply chain security

Image scanning in CI — block before it ships

Add trivy image --exit-code 1 --severity HIGH,CRITICAL $IMAGE after docker build. Trivy scans for CVEs. HIGH or CRITICAL vulnerabilities fail the build. Fix the image before it ever reaches the cluster.

Binary Authorization — only signed images in prod

GCP Binary Authorization requires images to be attested (signed) by your CI pipeline before GKE will run them. An image that didn't come from your pipeline cannot be deployed, even by someone with cluster credentials.

Distroless base images

Replace FROM node:20 with FROM gcr.io/distroless/nodejs20-debian12. No shell, no package manager, no utilities. A compromised pod has nowhere to go. Production containers should not have a shell.

↗ Interview questions — Terraform & Architecture

Senior / Staff · asked at Google, Cloudflare, Stripe
"Why would you choose Terraform over scripting infrastructure with the cloud CLI?"
The answer isn't "Terraform is better." It's about idempotency and state reconciliation. A bash script using gcloud is a one-way operation — if it fails halfway, you don't know what exists and what doesn't. Terraform maintains a state file mapping the real world to your configuration. Running it twice is safe. It also enables plan/apply separation — you can see exactly what will change before committing, critical for production. And every change becomes a reviewable PR with a diff, not a shell history entry.
Staff · system design · multiple environments
"You need to deploy the same app to 5 different GCP projects. How do you structure this?"
Terraform modules + directory-per-environment. Modules define reusable infra components. Each environment directory calls the same modules with environment-specific variables — region, tier, replica counts, HA settings. Adding a new environment is a new 20-line main.tf that calls existing modules. This is DRY applied to infrastructure: the module is the single source of truth for "how we build a GKE cluster." The environment files only capture "which values to use."
Principal / Architect · reliability
"Define RTO and RPO. What are realistic targets for Note Canva?"
RPO (Recovery Point Objective): maximum acceptable data loss, measured in time. "We can lose no more than 5 seconds of writes." This drives replication strategy.

RTO (Recovery Time Objective): maximum acceptable downtime after a failure. "The app must serve traffic again within 2 minutes."

For Note Canva: RPO of 5–10 seconds is achievable with async cross-region replication (typical lag). RTO of 2 minutes is achievable with automatic LB failover + pre-warmed secondary cluster. The bottleneck is Cloud SQL replica promotion at ~5 minutes — which is why you keep the secondary GKE cluster warm and serving reads from the replica immediately, before DB promotion completes.
Staff · real-time systems · WebSocket specifics
"Your realtime service uses WebSockets. How do you handle active connections during a region failover?"
WebSocket connections are stateful — they don't transparently reroute like HTTP. During failover, active connections to the failed region will die. The client must reconnect. Good client code handles this: exponential backoff, reconnect to the same document channel.

The key engineering decision: document state must live in the database, not in memory. When a client reconnects to the secondary region, it fetches current state from the replica and picks up from there. If you've built realtime with in-memory state as source of truth, a region failover will expose that gap. Design sessions to be reconstructable from persistent storage at any point.
Principal · the meta question · no right answer
"Limited bandwidth. Which do you tackle first: Terraform migration, multi-region, GitOps, or full observability?"
Order by risk reduction per unit of effort:

1. Observability first — you cannot safely change anything you can't measure. Set up basic metrics and alerting before any other infra work. Otherwise you'll be flying blind during every subsequent migration.

2. Terraform second — converts infra from "only one person knows how to rebuild this" to reviewable, repeatable, documented. Medium effort, huge risk reduction.

3. GitOps third — dramatically improves deployment safety and audit trail. Compounds well with Terraform.

4. Multi-region last — highest complexity, highest cost. Only justified when your SLO target requires it or customer data residency demands it. It will consume months of engineering time and permanently increase operational complexity. Don't build it until you have a hard business requirement.
The meta-skill all these questions test

Every question above tests the same skill: can you reason about tradeoffs under constraint? The interviewer doesn't care which tool you pick. They care that you can identify the relevant variables — team size, compliance requirements, reliability targets, operational complexity — weight them, and arrive at a defensible position. The candidate who says "Terraform Cloud is always better" fails. The candidate who says "it depends on these three factors, and here's how I'd evaluate them" passes.
